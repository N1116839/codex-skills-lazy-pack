# HTML Training Deck / HTML簡報_內部稽核

本 skill 用於製作「可閱讀、可互動、可稽核追溯」的 HTML 內部教育訓練簡報。它不是一般對外簡報，也不是只放大表格的資料頁。使用者的目的通常是：讓新人能快速了解 GDP 是什麼、為什麼要做、稽查重點是什麼、要準備哪些資料、為何要準備、怎麼做、什麼時間做。

## 核心原則

1. 一般簡報重點化；HTML 內部教育訓練要完整、扎實、可查、可互動。
2. 資料正確性優先於視覺效果。所有文件編號、週期、表單、部門權責與條件，必須回到來源文件。
3. 讀取本地資料時一律用 UTF-8 / Unicode-safe 方式，避免繁體中文亂碼。不要混用會造成亂碼的讀取方式。
4. 更新、開工、收工、踩坑點與第二大腦紀錄，必須用精確繁體中文撰寫。
5. learner-facing HTML 不顯示開發者設定、資料庫路徑、部署提醒、API key 說明或內部風險警語。
6. 如果同一資料有新版與舊版，需比對後以資料夾內最新版 SOP / WI / FR 內容為主；版次欄可不作為主要判斷依據。
7. 如果外部目錄、整理表或使用者提供資料與 SOP 原文有出入，必須明確告知使用者，讓使用者修正來源資料。

## 開工流程

1. 先讀專案接續文件、AGENTS.md、第二大腦專案頁與相關製作標準。
2. 盤點來源資料。若是 GDP 專案，需確認第一章至第八章資料夾內文件都已讀完或已建立完整索引。
3. 不只看檔名，要從 SOP/WI/FR 內容抽出：目的、權責、流程、表單、時間點、觸發條件、保存要求、稽查重點。
4. 先建立資料模型，再設計頁面。不要直接把未整理文字塞進 HTML。
5. 每次 substantial edit 前列出待辦與風險；完成後更新 README_Codex接續.md 與第二大腦。

## GDP 內部教育訓練專案規則

### 教材定位

- 品質手冊：說明「為何要做 GDP」，是 SOP 最高原則。內容應以品質系統、品質政策、風險管理、管理審查、稽核與持續改善為主。
- SMF 廠商清冊：說明「新勝醫藥怎麼做 GDP」，應呈現公司場所、組織、產品清冊、供應商、客戶、委外物流、儲存條件與作業範圍。
- 部門頁：各部門需分別呈現該部門要做哪些資料、何時做、怎麼做、用哪份表單、誰審核、異常時回到哪個流程。
- 年度行事曆：給「收資料的人」核對使用，可做互動式勾選、部門篩選、月份/頻率視圖、事件觸發視圖。

### 部門篩選固定名單

部門篩選只保留以下八個 learner-facing 部門：

- GDP權責人
- 管理藥師
- 品保
- 倉管
- 採購
- 業務
- 文管
- 人事

收件部門欄位可依 SOP 原文保留原權責部門，例如「倉管部」、「業務部」、「品管」等；但左側導覽與篩選主類別需使用上述固定名單。

### 週期與行事曆規則

1. 只有 SOP 明定固定週期，才列為正式週期項目。
2. 若時間節點沒有明定，不要為了湊完整而列出；但要確認相關文件已讀過。
3. 每季與每三個月視為同義週期，不可重複列出。
4. 重複時間點要合併，避免行事曆膨脹。
5. 發生時啟動項目需獨立於固定週期，例如偏差、CAPA、客訴、退回、疑似偽禁仿冒藥、回收、人員異動、重大環境變更。
6. 若有使用者提供的時程目錄，應用它作為索引參考，但必須回頭核對 SOP 原文；兩者有出入時以 SOP 為主並回報差異。
7. 年度資料收件行事曆不可只做大表格；若需表格，表格只是查詢層，主視覺應有時間軸、部門泳道、勾選清單、事件流程或互動篩選。

### 已知 GDP 內容提醒

下列事項在新勝 GDP 專案中曾被使用者指出，後續整理時必須重新核對 SOP 原文：

- 發生時：FR22-01 組織各職稱指派名單，人員有異動時需重新執行。
- 每日：採購部門應列印到貨驗收紀錄表。
- 運輸溫度紀錄：權責為倉管部。
- 物流門管控時間：15:00-15:30；收貨期間，進貨人員不得執行進貨作業。
- 每半年：溫控警報系統、門禁警報系統。
- 每三年與重大事件：需核對環境溫度測繪；重大環境變更時也可能需執行溫度測繪。
- 每三年曾被指出缺漏：WI10-01 SMF廠商基本資料、DP12-01 品質管理系統程序書、FR12-01 GDP權責主管任命書。實作時需重新讀文件確認。

## 視覺化與互動規則

當使用者給定資料時，Codex 需自行判斷以下事項，不要等使用者逐頁指定：

1. 哪些頁面需要 HTML 簡報底圖，並使用 imagegen / image2 生成符合內容的底圖。
2. 哪些段落需要扁平化圖標；可用 imagegen / image2 生成圖標後去背或裁切，再放入 HTML。
3. 哪些頁面需要互動環節，例如文字雲、選擇題、投票、簡答、Q&A、勾選清單、翻卡、情境判斷。
4. 哪些內容適合拖拉滑動、分層切換、時間軸、部門泳道、流程動畫、矩陣篩選或前後對照。
5. 稽查重點應做成互動式，而不是一整段靜態條列。可採用點選展開、題庫卡片、情境問答、證據鏈點選或部門篩選。
6. 依照文字內容產生對應視覺，不要只用同一種卡片或大表格套版。
7. HTML 內訓可比一般簡報更豐富，但仍要有清楚資訊架構，不可雜亂無章。


## 本專案沉澱規則：一次性整批修正

當使用者明確要求「一次做完、一次改正」時，處理 HTML 內訓教材不可只交付樣板或單頁示範。應在同一輪內完成同一批結構性問題，至少包含：資料模型、導覽架構、主要互動元件、截圖驗證與接續紀錄。若仍有未完成事項，必須列入專案待辦，而不是口頭帶過。

### 導覽與資訊架構規則

- 左側導覽需清楚分層，不可把不同用途的頁面混在同一組。
- 新勝 GDP HTML 內訓目前固定三大分組：
  - 文件主軸：品質手冊、SMF廠商清冊。
  - 部門分類：各部門分類、人事、GDP權責人、管理藥師、品保、倉管、採購、業務、文管。
  - 年度行事曆：年度資料收件行事曆。
- 導覽渲染應使用固定分組順序，避免資料陣列順序造成同一分類重複出現。
- 年度資料收件行事曆屬於收件人視角，不應放在部門分類底下。

### 稽查重點互動規則

- 稽查重點不可只做靜態條列或卡片矩陣。
- 預設做成互動核對面板：左側為可點選重點，右側顯示該點的核對說明。
- 每個稽查重點至少提醒三件事：
  1. 找得到對應 SOP 或表單。
  2. 紀錄完整、即時、正確、可追溯。
  3. 資料缺漏時知道由哪個部門補正。
- 後續可依內容升級為情境題、選擇題、簡答、翻卡、拖拉排序或部門篩選。

## Firebase Q&A 規則

當頁面需要 Q&A 保存：

- 使用 Firebase Firestore。
- 預設集合：`gdpTrainingQa/{sectionId}/items/{itemId}`。
- 頁面需支援 Firestore 與 localStorage fallback，但正式交付前必須驗證 Firestore 寫入、讀回與回覆保存。
- Q&A 狀態可顯示「已連接」或「本機暫存」，但不得顯示資料庫設定細節給學員。
- Firestore rules 至少需支援公開讀取、建立問題、只更新 replies。

## 禁止事項

- 不可把資料庫設定、部署步驟、API key 說明、GitHub Pages 風險提示顯示在學員頁面。
- 不可在沒有 SOP 依據時自行創造固定週期。
- 不可把「每季」與「每三月」重複列成兩種時間點。
- 不可把不屬於指定部門清單的項目放入部門篩選。
- 不可只靠檔名或目錄摘要判斷內容；必須回到 SOP/WI/FR 原文核對。
- 不可讓 HTML 頁面變成巨大難讀表格。
- 不可用亂碼內容更新 skill、第二大腦或交付頁。

## 驗證與收工

完成 HTML 內訓頁後至少要做：

1. HTML / JavaScript 語法檢查。
2. 以瀏覽器截圖檢查首頁與至少一個非預設部門頁。
3. 若有 Firebase Q&A，實際新增一筆測試問題、讀回、追加回覆，並確認 Firebase Console 有資料。
4. 檢查 learner-facing HTML 是否含開發者設定或不該顯示的風險提示。
5. 更新 README_Codex接續.md。
6. 更新第二大腦專案頁，使用精確繁體中文與一致 UTF-8 讀寫方式。
7. 收工時列出完成項目、未驗證項目、被阻擋原因、下一輪優先順序。

### 年度行事曆視覺與重複資訊規則

- 年度資料收件行事曆若每筆資料卡片已列「表單/紀錄、負責人、來源」，不要再另做「本頁相關 SOP」或完整文件清單，避免同一資訊重複三次。
- 若使用者質疑「只列出一些文件、沒有列全部文件」，需判斷頁面目的：收件行事曆只列有收件、啟動或核對責任的文件；完整 SOP 覆蓋狀況應放在接續紀錄或第二大腦，不塞回學員畫面。
- 資料收件核對台應優先用卡片式核對清單，呈現時間點、部門、資料名稱、啟動說明、表單/紀錄、負責人、來源。
- 「已完成 / 未完成」狀態應放在每筆資料卡片的來源或責任資訊旁，不另做一個脫離資料脈絡的「本月收件勾選」區。
- 「依頻率分組」若已與核對卡片重複，應降級為摘要，只顯示頻率數量與涉及部門數，不重複列明細。
- 關鍵設備行事曆可沿用年度行事曆的資訊架構：左側頻率選單，右側條列設備、作業、表單、來源與負責人。
